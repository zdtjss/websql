# WebSQL 数据权限体系全面梳理

## 一、整体架构概览

WebSQL 的权限体系采用 **四层分级模型**，围绕"连接(Conn) → Schema → 表(Table) → 列(Column)"逐级细化，并遵循 **"最具体优先"原则**（即当存在低级别权限配置时，高级别权限自动降级）。整个权限系统分为后端和前端两条线，**后端是权限执行的权威，前端仅做展示层面的过滤**。

### 技术栈

- 后端：Go + Gin + SQLite/MySQL
- 前端：Vue 3 + Element Plus + TypeScript
- AI Agent：基于 LLM 的 SQL 生成与权限审核

## 二、核心数据模型

### 2.1 数据库表结构

```sql
-- 角色表
CREATE TABLE t_role (
    id VARCHAR(36) PRIMARY KEY,
    name VARCHAR(64),
    view_classic INTEGER DEFAULT 0   -- 是否允许使用经典视图
);

-- 权限表（四级层级）
CREATE TABLE t_power (
    id VARCHAR(36) PRIMARY KEY,
    role_id VARCHAR(36),             -- 关联角色
    conn_id VARCHAR(36),             -- 数据库连接ID
    schema_name VARCHAR(128),        -- 库名（schema级及以上为null）
    table_name VARCHAR(128),         -- 表名（table级及以上为null）
    column_name VARCHAR(128),        -- 列名（column级为具体列名，其余为null）
    power_level VARCHAR(32)          -- 权限层级：conn / schema / table / column
);

-- 用户-角色关联表
CREATE TABLE t_user_role (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36),
    role_id VARCHAR(36)
);

-- 连接配置表
CREATE TABLE t_conn (
    id VARCHAR(36) PRIMARY KEY,
    db_type VARCHAR(64),
    parent_id VARCHAR(36),           -- 所属目录ID
    name VARCHAR(64),
    user VARCHAR(64),
    pwd VARCHAR(256),
    url VARCHAR(512),
    db_schema VARCHAR(128),
    db_version VARCHAR(64)
);

-- 目录树表
CREATE TABLE t_tree (
    id VARCHAR(36) PRIMARY KEY,
    label VARCHAR(64),
    parent VARCHAR(36)
);
```

### 2.2 Go 数据结构

```go
// 角色
type Role struct {
    Id          string         `json:"id"`
    Name        string         `json:"name"`
    PowerList   []*PowerDetail `json:"powerList"`
    ViewClassic int            `json:"viewClassic" db:"view_classic"`
}

// 角色保存请求（增量更新）
type RoleSave struct {
    Id          string         `json:"id"`
    Name        string         `json:"name"`
    AddPowers   []*PowerDetail `json:"addPowers"`   // 新增的权限
    DelPowers   []*PowerDetail `json:"delPowers"`   // 删除的权限
    ViewClassic int            `json:"viewClassic"` // 经典视图开关
}

// 权限详情
type PowerDetail struct {
    Id         string  `json:"id"`
    RoleId     string  `json:"roleId" db:"role_id"`
    ConnId     string  `json:"connId" db:"conn_id"`
    ConnName   *string `json:"connName,omitempty" db:"conn_name"`
    SchemaName *string `json:"schemaName,omitempty" db:"schema_name"`
    TableName  *string `json:"tableName,omitempty" db:"table_name"`
    ColumnName *string `json:"columnName,omitempty" db:"column_name"`
    Level      string  `json:"level" db:"power_level"`
}

// 权限树节点
type PermissionNode struct {
    Id       string            `json:"id"`        // 格式: connId[::schema][::table][::column]
    Label    string            `json:"label"`
    Type     string            `json:"type"`      // dir / conn / schema / table / view / column
    Level    string            `json:"level"`     // dir / conn / schema / table / column
    ParentId string            `json:"parentId,omitempty"`
    Checked  bool              `json:"checked,omitempty"`
    Data     map[string]any    `json:"data,omitempty"`
    Children []*PermissionNode `json:"children"`
}

// 用户权限摘要（存储在 session/cache 中）
type UserPower struct {
    UserId string
    Power  []string   // 有权限的 connId 列表
}

// AI Agent 权限范围
type PermissionScope struct {
    UserID, ConnID, SchemaName, IsRemote string
    HasFullConnAccess   bool
    HasFullSchemaAccess bool
    AllowedTables       map[string]bool
    AllowedColumns      map[string]map[string]bool
}

// Schema 引用（前端选择后传递给后端）
type SchemaRef struct {
    ConnID string `json:"connId"`
    Schema string `json:"schema"`
}
```

### 2.3 四级权限层级

| 级别 | Level 值 | 含义 | 数据库字段 |
|------|----------|------|-----------|
| 连接级 | `conn` | 该连接下所有库表列完整访问 | 仅 `conn_id` |
| Schema级 | `schema` | 该 Schema 下所有表列完整访问 | `conn_id` + `schema_name` |
| 表级 | `table` | 该表所有列完整访问 | `conn_id` + `schema_name` + `table_name` |
| 列级 | `column` | 仅指定列可访问 | `conn_id` + `schema_name` + `table_name` + `column_name` |

### 3.2 前端权限配置界面（RolePermission.vue）

**核心状态管理**：

- `explicitKeys`：用户手动勾选的权限 key，只有这些会保存到后端
- `implicitKeys`：因下级有权限而视觉标记的父级 key，不保存
- `baselineCheckedKeys`：保存时的对比基准（来自 role.powerList）
- `serverCheckedKeys`：API 返回的 checked=true 状态（含向上传播）
- `uncheckedKeys`：用户显式取消勾选的 key

**权限 key 格式**：`connId[::schemaName][::tableName][::columnName]`

例如：
- 连接级：`abc123`
- Schema级：`abc123::mydb`
- 表级：`abc123::mydb::users`
- 列级：`abc123::mydb::users::email`

**导航机制**：面包屑导航，逐级深入：
```
连接(conn) → Schema → 表(table) → 列(column)
```

点击节点名称进入下一级，勾选复选框授予权限。

**保存逻辑**：

1. 按当前编辑层级筛选 explicitKeys 和 baselineCheckedKeys
2. **Schema 权限过滤**：如果某 schema 下有 table/column 子级权限，则该 schema 级权限不发送（因为子级权限更具体，schema 级多余）
3. 计算增量：`addPowers` = 新增的权限，`delPowers` = 删除的权限
4. 发送 `POST /saveRole` 请求

> 涉及文件：[RolePermission.vue](web-src/src/views/RolePermission.vue)

### 3.3 后端权限树 API（GetPermissionTree）

根据 `level` 参数返回不同层级的权限树：

| level | 函数 | 说明 |
|-------|------|------|
| conn | `getConnTree` | 返回目录+连接节点，目录作为父节点包含子连接 |
| schema | `getSchemaTree` | 返回指定连接下的所有 schema |
| table | `getTableTree` | 返回指定 schema 下的所有表 |
| column | `getColumnTree` | 返回指定表下的所有列 |

每个节点都包含 `checked` 字段，表示当前角色是否已有该权限。**向上传播**在查询时实现：
- `getRoleConnPermissions`：同时查 conn 级和 schema/table/column 级权限
- `getRoleSchemaPermissions`：同时查 schema 级和 table/column 级权限
- `getRoleTablePermissions`：同时查 table 级和 column 级权限

> 涉及文件：[admin.go](web-api/admin/admin.go) `GetPermissionTree` 函数

### 3.4 角色保存（增量更新）

使用事务进行增量更新：

1. 保存或更新角色基本信息（含 view_classic）
2. 批量插入新增权限 (`insertPowers`)：使用批量 SQL `INSERT INTO t_power (...) VALUES (...), (...), (...)`
3. 批量删除权限 (`deletePowers`)：按层级分组优化
   - conn 级：`DELETE WHERE conn_id IN (?, ?, ?) AND power_level = 'conn'`
   - schema 级：按 connId 分组批量删除
   - table/column 级：逐条精确删除

> 涉及文件：[admin.go](web-api/admin/admin.go) `SaveRole`、`insertPowers`、`deletePowers` 函数

## 四、Schema 选择与使用（大模型对话中指定的 Schema）

### 4.1 前端 Schema 选择器（App.vue）

**核心状态变量**：

```javascript
const selectedSchemas = ref([])       // 选中的 schema 值列表，格式为 "connId::schemaName"
const tableList = ref([])             // 当前 schema 下的表列表
const connList = ref([])              // 连接列表
const connSchemaList = ref([])        // 连接+schema 结构列表
```

**是否显示 Schema 选择器**：

```javascript
const shouldShowSchemaSelector = computed(() => {
  if (schemasLoading.value) return true
  if (connSchemaList.value.length > 1) return true
  if (connSchemaList.value.length === 1) {
    const conn = connSchemaList.value[0]
    const schemaCount = (conn.schemas || []).length
    if (schemaCount > 1) return true
  }
  return false
})
```

逻辑：只有单连接且单 schema 时才隐藏选择器。

**Schema 树构建**：`buildSchemaTree()` 将 `connSchemaList` 转为 `el-tree-select` 支持的树结构：
- 单 schema 连接：折叠为叶子节点，`value = "connId::schemaName"`
- 多 schema 连接：连接作为目录节点（disabled），schema 作为可选子节点
- 支持目录分组（dirName）

**Schema 值解析**：

```javascript
function parseSchemaValue(value) {
  const idx = value.indexOf('::')
  if (idx === -1) return null
  return { connId: value.substring(0, idx), schema: value.substring(idx + 2) }
}
```

**发送消息时的 Schema 传递**：

```javascript
const schemas = buildRequestSchemas()
const body = {
  sessionId: sessionId.value,
  connId: primaryConnId,      // 主连接ID（向后兼容）
  schema: primarySchema,       // 主 schema（向后兼容）
  schemas,                     // 完整的 SchemaRef 数组
  question: messageContent,
  tableContext: selectedTables.value,
  modelId: selectedModel.value,
}
```

关键点：同时发送 `connId`/`schema`（向后兼容）和 `schemas` 数组（多 schema 模式）。

> 涉及文件：[App.vue](web-src/src/App.vue)

### 4.2 提示词编辑器 Schema 选择（PromptEditDialog.vue）

与 App.vue 类似的 Schema 选择逻辑，独立维护 `selectedConnSchema`、`connTree` 等状态。支持将选中的 connSchema 和 tables 传递给 AI 对话。

### 4.3 后端请求解析（handler.go）

**ChatRequest 结构**：

```go
type ChatRequest struct {
    SessionID    string      `json:"sessionId"`
    UserID       string      `json:"userId"`
    ConnID       string      `json:"connId"`
    Schema       string      `json:"schema"`
    Schemas      []SchemaRef `json:"schemas,omitempty"` // 多 schema 模式
    Question     string      `json:"question"`
    TableContext []string    `json:"tableContext"`
    ...
}
```

**resolveRequestParams** 是 Schema 解析的核心入口：

1. `connID` 优先从 `req.ConnID` 取，为空时从 `req.Schemas[0].ConnID` 取
2. `dbSchema` 优先从 `req.Schemas[0].Schema` 取，其次从 `req.Schema` 取
3. 用 `connID` + `dbSchema` 构建权限范围 `PermissionScope`

**prepareSession**：将 `req.Schemas` 和 `req.TableContext` 保存到会话上下文，用于历史会话恢复。

**createAgent**：将 schemas 传递给 Agent 工厂创建/缓存 Agent。

> 涉及文件：[handler.go](web-api/ai/agent/v2/handler.go)

### 4.4 Agent 创建与 Schema 注入

**Agent 缓存 Key**：`connID::dbSchema::userID`

注意：缓存 Key 只用了 `connID + dbSchema + userID`，**没有包含完整的 schemas 数组**。这意味着如果用户切换了多 schema 选择但主连接和主 schema 不变，会复用缓存的 Agent。

**SQLAgent 创建中 Schema 的使用**：

1. **构建工具**：`buildTools(ctx, connID, dbType, dbSchema, schemas, auditCtx)` — schemas 传入工具构建
2. **构建权限 Agent**：如果 `scope.IsRemote && !scope.HasFullConnAccess`，创建 PermissionAgent
3. **构建系统提示词**：`buildSystemPrompt(connID, dbType, dbSchema, dbVersion, nil, scope, schemas)`

> 涉及文件：[agent.go](web-api/ai/agent/v2/agent.go)、[agent_factory.go](web-api/ai/agent/v2/agent_factory.go)

### 4.5 工具中的 Schema 路由逻辑

**连接查找映射**：

```go
func buildConnLookup(schemas []SchemaRef) map[string]string {
    connLookup := make(map[string]string)
    for _, s := range schemas {
        if s.ConnID != "" && s.Schema != "" {
            key := strings.ToUpper(s.Schema)
            connLookup[key] = s.ConnID
        }
    }
    return connLookup
}
```

构建 `schema名(大写) → connID` 的映射表，用于工具调用时根据 schema 名路由到正确的数据库连接。

**各工具的 Schema 路由**：

| 工具 | Schema 路由方式 |
|------|----------------|
| `query_data` | `resolveConnID()` 解析目标连接，失败后自动 `tryAlternativeConn` 路由 |
| `exec_sql` | 同 query_data |
| `get_table_schema` | `splitSchemaTable` 拆分 schema.table，通过 `schemaConnMap` 查找连接 |
| `list_tables` | 支持 `connId` 参数路由到不同连接 |

**裸表名自动补全**：`qualifyBareTableNames` 自动为裸表名添加 schema 前缀。

> 涉及文件：[tools.go](web-api/ai/agent/v2/tools.go)

### 4.6 系统提示词中的 Schema 注入

`buildDynamicPromptPart` 根据 schemas 数量生成不同的上下文信息：

**单 schema 模式**：
```
当前环境 — 数据库：mysql，版本：8.0，Schema：mydb
```

**多 schema 模式**：
```
**多 Schema 上下文**（按数据库连接分组，相同连接内的 schema 可直接 JOIN）：
  🔗 连接 abc123 (mysql)：
    - Schema: order_db
    - Schema: product_db
  🔗 连接 def456 (postgresql)：
```

**跨库操作规则**（多 schema 时注入）：
- 同连接 schema 可 JOIN，不同连接必须分步查询
- `query_data`/`exec_sql` 的 `connId` 参数用法（填 Schema 名自动路由）
- 写操作同样受连接限制
- 数据来源标注要求

## 五、权限使用/校验逻辑

### 5.1 最具体优先原则（核心安全规则）

当同一连接下同时存在高级别与低级别权限时，以最具体的权限配置为准。即：若存在 table/column 级权限，则 conn/schema 级权限自动降级，不得越权放行。

该规则在多处代码中统一实现：

| 位置 | 函数 | 说明 |
|------|------|------|
| admin.go | `checkPower` | `hasConnPermission && !hasTableOrColumnForSchema` 才返回 true |
| conn_config.go | `checkPowerByParam` | 同样的逻辑 |
| sql_analyzer.go | `GetTableColumnAccess` | 列级权限查询 |
| permission.go | `BuildPermissionScope` | AI Agent 权限范围构建 |

### 5.2 登录认证流程

支持三种登录方式：

1. **密码登录** (`pwd`)：通过 `findByLoginName` 查找用户，MD5+salt 校验密码
2. **生物识别** (`bio`)：通过 `findByBio` 查找指纹/面容对应的用户
3. **Token登录** (`token`)：通过外部系统验证 token，调用 `cfg.OutterUser` 接口获取用户信息

登录成功后：
- 生成安全随机 token（`utils.SecureRandomToken()`）
- 查询用户权限 `findUserPower(userId)` 获取 connId 列表
- 将 `UserPower` 和 `User` 存入 store（带 `USER:POWER:` 前缀）
- 写入本地内存缓存 `authCache`（10秒 TTL）
- 返回 `{id, name, isAdmin, authentication: token}`

> 涉及文件：[login.go](web-api/admin/login.go)

### 5.3 中间件权限检查

**全局中间件链**：

```
CustomRecovery → Gzip → LoginRateLimit → APIRateLimit → CircuitBreaker → AuthMiddleware → hostCheck → CORS
```

**认证中间件 `AuthMiddleware`**：

- 跳过路径：`/api/login`, `/api/logout`, `/api/healthCheck`, `/api/sysMode`, `/assets`
- 从 `Authorization` header 获取 token
- 调用 `admin.GetCachedUserAndPower(authorization)` 验证 token
- 无效返回 401，有效则将 `currentUser`/`userId`/`userPower` 写入 gin context

**管理员权限检查 `CheckAdminPower`**：

- 非远程模式跳过
- 仅 `config.AdminId` 用户可通过
- 用于：SaveTree, DelTreeNode, ListDirTree, ConnBaseTree, SaveConn, TestDbConn, DelConn, ListConn2, SaveRole, DelRole, SaveUser, DelUser 等管理接口

**限流中间件**：

- 登录限流：每 IP 0.5 req/s，burst 5
- API限流：普通 API 50 req/s burst 100；AI 接口 2 req/s burst 5
- 使用分片（64 shards）+ 令牌桶算法

> 涉及文件：[router.go](web-api/router.go)、[ratelimit.go](web-api/ratelimit.go)

### 5.4 SQL 执行权限检查（经典模式）

`ExecSQL` 函数的权限检查流程：

```
1. SQL长度限制（1MB）
2. 获取 Authorization 和 connId
3. 获取数据库连接（GetConn 内部做连接级权限检查）
4. 当 schema 为空时，从数据库连接获取实际 schema
5. AnalyzeSQL(sqlStr, schema) → 分析SQL操作类型和涉及的表
6. CheckAnalysisPermission(analysis, connId, authorization) → 表级权限检查
7. 多条SQL时，逐条重复步骤5-6
8. 权限通过后执行SQL
```

**SQL 分析器** `AnalyzeSQL` 解析 SQL 并返回 `SQLAnalysis`：

```go
type SQLAnalysis struct {
    OperationType string       // SELECT, INSERT, UPDATE, DELETE, DDL, UNKNOWN
    ReadTables    []TableRef   // 读操作涉及的表
    WriteTables   []TableRef   // 写操作涉及的表
    WriteColumns  []ColumnRef  // 写操作涉及的列
    OriginalSQL   string
}
```

支持解析：INSERT INTO ... (col1, col2)、UPDATE ... SET col1=v1, col2=v2、DELETE FROM、DDL（ALTER/DROP/CREATE/TRUNCATE/RENAME）、CTE（WITH 子句）。

**SQL 权限校验** `CheckAnalysisPermission` 核心逻辑：

1. 检查读表权限：遍历 `analysis.ReadTables`，调用 `GetTableAccessDowngraded` 检查
2. 检查写表权限：遍历 `analysis.WriteTables`，调用 `GetTableAccessDowngraded` 检查
3. 无权限则标记 `Allowed = false`

**注意**：经典模式使用 `GetTableAccessDowngraded`，将列级权限降级为表级权限（`AccessColumn` → `AccessFull`），即经典模式下不区分列级权限。

> 涉及文件：[sql_exec.go](web-api/sql_exec.go)、[sql_analyzer.go](web-api/admin/sql_analyzer.go)、[sql_permission.go](web-api/admin/sql_permission.go)

### 5.5 表列级权限查询（核心判断函数）

`GetTableColumnAccess` 是权限判断的核心函数：

```
1. 非远程模式 → AccessFull
2. userPower 为 nil → AccessNone
3. userId == AdminId → AccessFull
4. 查询 findUserPowerDetails
5. 遍历权限记录：
   - conn级 → hasConnLevel
   - schema级且匹配 → hasSchemaLevel
   - table级且匹配 → hasTableLevel + hasTableOrColumnForSchema
   - column级且匹配 → 收集 allowedCols + hasTableOrColumnForSchema
6. 判断结果（最具体优先）：
   - hasConnLevel && !hasTableOrColumnForSchema → AccessFull
   - hasSchemaLevel && !hasTableOrColumnForSchema → AccessFull
   - hasTableLevel → AccessFull
   - len(allowedCols) > 0 → AccessColumn
   - 否则 → AccessNone
```

> 涉及文件：[sql_analyzer.go](web-api/admin/sql_analyzer.go) `GetTableColumnAccess` 函数

### 5.6 数据库操作权限检查

| 操作 | 权限检查函数 | 说明 |
|------|-------------|------|
| 导出 (ExportXlsx) | `CheckTablePermission` | 调用 `checkSchemaAccess` + `checkTableAccess` |
| 导入 (ImportXlsx) | `CheckTableWritePermission` | 同上（当前未区分读写权限） |
| 表元数据 (TableOptions等) | `CheckTablePermission` | 同上 |

> 涉及文件：[export.go](web-api/export.go)、[import.go](web-api/import.go)、[db_operate.go](web-api/admin/db_operate.go)

### 5.7 树形结构权限过滤

`ShowTree` 根据节点类型调用不同的过滤函数：

| 节点类型 | 过滤函数 | 说明 |
|---------|---------|------|
| dir | `filterDirTreeWithPermission` | 目录节点，根据子连接权限向上传播 |
| conn | `filterConnsWithPermission` | 连接节点，SQL追加权限过滤 |
| schema | `filterSchemasWithPermission` | Schema节点，根据权限详情过滤 |
| table | `filterTablesWithPermission` | 表节点，根据权限详情过滤 |
| column | `listColumns` | 列节点，根据表级权限过滤 |

**目录权限传播**：`filterDirTreeWithPermission` 实现了目录权限的**向上传播**：
1. 查询用户有权限的所有连接
2. 找到连接所属的目录
3. 从目录向上递归传播，使父目录也可见

**连接级权限过滤**：`filterConnsWithPermission` 使用 `appendPmsn` 函数在 SQL 中追加 `WHERE id IN (?, ?, ...)` 条件，只返回用户有权限的连接。

**Schema/表级过滤**遵循相同的逻辑：
- 非远程模式：返回全部
- 管理员：返回全部
- conn级权限且无下级配置：返回全部
- schema级权限且无下级配置：返回全部
- 有table/column级配置：只返回匹配的项

> 涉及文件：[tree_permission.go](web-api/admin/tree_permission.go)、[conn_config.go](web-api/admin/conn_config.go)

### 5.8 AI Agent 权限体系

#### 5.8.1 PermissionScope 构建

`BuildPermissionScope(userId, connId, schemaName)` 是 AI 权限的核心构建函数：

1. 非远程模式直接跳过权限检查
2. 查询用户权限记录 `admin.FindUserPowerDetails(userId)`
3. 按 `connId` 过滤，按权限级别分类：
   - `conn` 级：`HasFullConnAccess = true`
   - `schema` 级：`HasFullSchemaAccess = true`（需匹配 schemaName）
   - `table` 级：记录到 `AllowedTables`
   - `column` 级：记录到 `AllowedColumns`

**关键点**：`schemaName` 参数用于过滤权限记录，只有匹配该 schema 的权限才生效。

#### 5.8.2 PermissionMiddleware（AI Agent 权限中间件）

拦截所有工具调用，按工具类型执行不同的权限检查：

| 工具 | 检查方式 |
|------|---------|
| `get_table_schema` | 过滤无权限的表 + DDL列级过滤 |
| `list_tables` | 后过滤（postFilter）移除无权限表 |
| `query_data` | AI Agent判断 + 结果集列过滤 |
| `exec_sql` | AI Agent判断 + 程序化降级检查 |
| `export_*` | AI Agent判断 + 程序化降级检查 |
| `import_data` | 表级 + 列级写权限检查 |
| 其他工具 | 直接放行 |

**双重防线设计**：
1. **第一道**：AI Permission Agent 判断（通过 LLM 分析 SQL 是否越权）
2. **第二道**：结果集过滤（`FilterResultColumns` 兜底，移除未授权列的数据）

#### 5.8.3 Permission Agent（AI 权限审核 Agent）

独立的 LLM Agent，专门用于权限审核。它有两个工具：
- `get_table_structure`：获取表结构信息
- `get_user_permissions`：获取用户权限配置

输出 `PermDecisionOutput`：

```go
type PermDecisionOutput struct {
    Allowed       bool     `json:"allowed"`
    DeniedTables  []string `json:"deniedTables"`
    DeniedColumns []string `json:"deniedColumns"`
    Reason        string   `json:"reason"`
}
```

系统提示词中注入了：
- 数据库类型、默认 Schema、用户 ID、连接 ID
- 权限层级规则（最具体优先原则）
- SELECT * 拦截规则
- 系统表放行规则

#### 5.8.4 权限描述注入提示词

`PermissionScope.DescribeForPrompt()` 将权限范围注入系统提示词：

```
## 数据权限（最高优先级）
绝对禁止使用、提及任何未授权表的信息。

表级权限（可访问所有字段）：orders, products

字段级权限 - 表 `users`：仅允许 [id, name]，其他字段禁止使用
```

#### 5.8.5 危险SQL审批中间件

`DangerousSQLApprovalMiddleware` 拦截 `exec_sql` 工具中的写操作（DROP/TRUNCATE/DELETE/ALTER/CREATE/REPLACE/INSERT/UPDATE/MERGE），强制用户确认：

- 阶段1（首次调用）：安全SQL直接放行，危险SQL中断并返回确认信息给前端
- 阶段2（从中断恢复）：用户确认后放行，拒绝则返回拒绝信息

#### 5.8.6 Permission Agent 缓存

缓存 key = `connID::dbSchema::userID`，TTL = 5 分钟，每 2 分钟清理过期条目。当 AI 配置变更时（cfgHash 不同）自动重建。

> 涉及文件：[permission.go](web-api/ai/agent/v2/permission.go)、[permission_agent.go](web-api/ai/agent/v2/permission_agent.go)、[permission_agent_tools.go](web-api/ai/agent/v2/permission_agent_tools.go)、[permission_agent_cache.go](web-api/ai/agent/v2/permission_agent_cache.go)、[middleware.go](web-api/ai/agent/v2/middleware.go)

### 5.9 前端权限控制

**路由守卫**：仅对 `requiresClassicView` 路由做守卫，调用 `/canUseClassicView` 接口检查角色是否允许使用经典视图。

**Schema Store**：前端不直接做权限判断，仅缓存后端已过滤的表结构数据。权限过滤完全由后端在 API 层完成。

> 涉及文件：[router/index.ts](web-src/src/router/index.ts)、[stores/dbSchema.ts](web-src/src/stores/dbSchema.ts)

## 七、关键安全设计要点

1. **非远程模式免权限**：`config.Cfg.IsRemote == false` 时，几乎所有权限检查都跳过，适合本地单机使用
2. **管理员豁免**：`userId == config.AdminId` 在所有权限检查中直接通过
3. **最具体优先原则**：当存在 table/column 级权限时，conn/schema 级权限自动降级
4. **经典模式列级降级**：`GetTableAccessDowngraded` 将 `AccessColumn` 降为 `AccessFull`，经典视图不区分列级权限
5. **AI 双重防线**：Agent 判断 + 结果集过滤，确保即使 LLM 判断失误也不会泄露数据
6. **默认拒绝**：AI 权限中间件中，SQL 为空或解析失败时一律拒绝
7. **密码安全**：MD5 + 固定 salt，数据库密码 AES 加密存储
8. **Token安全**：使用 `utils.SecureRandomToken()` 生成，支持本地缓存 + 分布式 store

## 八、数据库连接未指定 Schema 时的专项评估

### 8.1 核心认知：连接 db_schema 为空 ≠ 运行时 schema 为空

数据库连接（`t_conn` 表）的 `db_schema` 字段可能为 NULL/空字符串（如 MySQL 连接未在 URL 中指定默认数据库、新建连接尚未自动检测 schema 等）。但**这并不意味着运行时 schema 为空**，因为在正常使用流程中，用户通过界面选择 schema，运行时的 schema 是明确的：

| 使用场景 | Schema 来源 | 是否可能为空 |
|---------|------------|------------|
| **经典模式 - SQL 执行** | 用户点击树中 schema 节点 → `addTab(node)` 中 `schema = node.data.label` → 传入 `POST /execSQL` | ❌ 不会。schema 来自用户点击的树节点 |
| **经典模式 - 树展开** | 展开 conn 节点获取 schema 列表（schema 名通过 `key` 参数传递）；展开 schema 节点获取表列表；展开 table 节点获取列（`schema = parent.label`） | ❌ 不会。每级展开都传递了明确的参数 |
| **AI 对话** | 用户在 schema 选择器中选择 → `sendMessage` 强制校验 `schemas.length > 0` → 后端收到 `req.Schemas[0].Schema` | ❌ 正常不会。前端拦截了空选择 |
| **权限配置** | 管理员通过面包屑导航逐级点击 → 每级传递 connId/schemaName/tableName | ❌ 不会。UI 逐级导航，参数明确 |
| **后端兜底** | `ExecSQL` 中 `schema == ""` 时执行 `SELECT DATABASE()` 获取；`listColumns` 中调用 `getCurrentSchema(dc)` | 兜底逻辑，正常流程不应触发 |

**结论：在正常使用流程中，运行时 schema 始终是明确的，不会出现空 schema 的情况。** 连接的 `db_schema` 字段只是一个默认值/元数据，实际使用时由用户选择覆盖。

### 8.2 唯一的边缘场景：SchemaRef.schema 为空字符串

存在一个特定的边缘场景，可能导致运行时 schema 为空：

**触发条件**（需同时满足）：
1. 连接的 `db_schema` 字段为 NULL（未配置）
2. 该连接的 schema 列表查询返回空（如连接失败无法查询 `SHOW SCHEMAS`）
3. 前端自动选中该连接（单连接单 schema 自动选中逻辑），产生 `connId::` 节点值
4. `parseSchemaValue("connId::")` 返回 `{connId: "xxx", schema: ""}`

**数据流**：

```
t_conn.db_schema = NULL 且 SHOW SCHEMAS 查询失败/返回空
    │
    ▼ 前端 buildSchemaTree()
    │  schemas 数组为空 → singleSchema = item.dbSchema || '' = ''
    │  节点 value = "connId::"
    │
    ▼ 自动选中（shouldShowSchemaSelector 隐藏选择器时）
    │  selectedSchemas = ["connId::"]
    │
    ▼ buildRequestSchemas()
    │  parseSchemaValue("connId::") → {connId: "xxx", schema: ""}
    │  schemas.length === 1 → 不被前端拦截
    │
    ▼ 后端 resolveRequestParams()
    │  req.Schemas[0].Schema == "" → 不覆盖 dbSchema
    │  GetDBInfo() → dbSchema = "" (连接配置也为空)
    │  最终 dbSchema = ""
    │
    ▼ BuildPermissionScope(userId, connID, "") → 权限范围可能不正确
```

**这个场景在实际中很少发生**，因为：
- 保存/测试连接时会自动调用 `getDbVersionAndSchema` 检测 schema
- 即使检测失败，只要数据库连接正常，`SHOW SCHEMAS` 通常能返回结果
- 只有在连接本身有问题（无法查询元数据）时才会出现

### 8.3 各模块对空 Schema 的防御性处理

虽然正常运行时 schema 不为空，但代码中确实存在对空 schema 的防御性处理。以下梳理各模块的行为，评估其**防御性代码的合理性**：

#### 8.3.1 经典模式后端兜底

| 函数 | 空 schema 兜底逻辑 | 评价 |
|------|-------------------|------|
| `ExecSQL` | `SELECT DATABASE()` / `SYS_CONTEXT` / `"main"` | ✅ 合理，确保权限检查能正常工作 |
| `listColumns` | `getCurrentSchema(dc)` | ✅ 合理，与 ExecSQL 一致 |
| `ListTableNames` | 同上 | ✅ 合理 |

#### 8.3.2 权限校验函数的空 schema 分支

| 函数 | 空 schema 行为 | 评价 |
|------|---------------|------|
| `checkPowerByParam` | 只认 conn 级权限，其余失效 | ⚠️ 防御性策略：空 schema 时缩小权限，安全侧倒 |
| `checkPower` | `*p.SchemaName == ""` 无法匹配，同上 | ⚠️ 同上，安全侧倒 |
| `GetTableColumnAccess` | 同上 | ⚠️ 同上 |
| `BuildPermissionScope` | `schemaName == ""` 时放宽匹配，所有权限聚合 | ⚠️ **与上述函数策略相反**，向不安全侧倒 |

`checkPowerByParam` 的逻辑：

```go
if param.SchemaName == "" {
    // 空 schema 时只检查 conn 级权限
    for _, power := range powerDetails {
        if power.ConnId == param.ConnId && power.Level == "conn" {
            return true
        }
    }
    return false  // 没有 conn 级权限则拒绝
}
```

`BuildPermissionScope` 的逻辑：

```go
case "schema":
    if schemaName == "" || pSchema == schemaName {
        hasSchemaPerm = true  // 空 schema 时任何 schema 级权限都匹配
    }
case "table":
    if (schemaName == "" || pSchema == schemaName) && pTable != "" {
        scope.AllowedTables[pTable] = true  // 空 schema 时任何 table 级权限都匹配
    }
```

**关键差异**：
- `checkPowerByParam`：空 schema → **收紧**（只有 conn 级生效）— 安全侧
- `BuildPermissionScope`：空 schema → **放宽**（所有权限跨 schema 聚合）— 不安全侧

#### 8.3.3 权限树的空 schema 处理

| 函数 | 空 schema 行为 | 评价 |
|------|---------------|------|
| `GetPermissionTree` | table/column 级返回空数组 | ⚠️ 防御性策略，但正常流程不会触发（管理员必须先选 schema 才能进入下级） |

#### 8.3.4 AI 工具层的空 schema 处理

| 函数 | 空 schema 行为 | 评价 |
|------|---------------|------|
| `buildDynamicPromptPart` | 显示 "Schema：" | ⚠️ 提示词不完整，但正常流程不会触发 |
| `NewSchemaFunc` | 使用不带 schema 前缀的 SQL | ✅ 依赖连接默认数据库，MySQL 下可工作 |
| `NewListTablesFunc` | 使用 `DATABASE()` / `USER_TAB_COMMENTS` | ✅ 合理兜底 |
| `qualifyBareTableNames` | 跳过限定 | ✅ 合理，裸表名不加前缀也能工作 |

### 8.4 风险评估（修正后）

| 风险等级 | 问题 | 说明 |
|---------|------|------|
| **🟡 中** | `BuildPermissionScope` 空 schema 时放宽匹配 | 仅在边缘场景触发，但策略与经典模式不一致，若触发则权限放大 |
| **🟡 中** | 前端 `parseSchemaValue("connId::")` 产生空 schema 的 SchemaRef | 仅在连接无法查询 schema 列表时触发，但一旦触发会绕过前端空选择拦截 |
| **🟢 低** | `checkPowerByParam` 空 schema 时只认 conn 级权限 | 防御性策略，安全侧倒，正常流程不触发 |
| **🟢 低** | `GetPermissionTree` 空 schema 时返回空数组 | 正常流程不触发（管理员必须选 schema） |
| **🟢 低** | AI 提示词中 Schema 名为空 | 正常流程不触发 |

### 8.5 改进建议

1. **统一空 schema 处理策略**：`BuildPermissionScope` 中 `schemaName == ""` 时的放宽匹配逻辑应与 `checkPowerByParam` 保持一致（安全侧倒），或者改为动态获取当前 schema。推荐方案：
   ```go
   // 在 BuildPermissionScope 开头，如果 schemaName 为空，尝试从数据库获取
   if schemaName == "" {
       conn := GetConnNoCheck(connId)
       schemaName = getCurrentSchema(conn)
   }
   ```

2. **前端 `parseSchemaValue` 增加空 schema 校验**：当解析出的 schema 为空字符串时，应过滤掉该条目或提示用户，避免发送 `SchemaRef.Schema == ""` 的请求：
   ```javascript
   function buildRequestSchemas() {
     return selectedSchemas.value
       .map(v => parseSchemaValue(v))
       .filter(p => p && p.schema !== '')  // 过滤空 schema
       .map(p => ({ connId: p.connId, schema: p.schema }))
   }
   ```

3. **连接保存时强制检测 schema**：在 `SaveConn` 时，如果 `db_schema` 为空，自动调用 `getDbVersionAndSchema` 填充，从源头减少空 schema 的出现概率。

4. **`resolveRequestParams` 增加兜底**：当最终 `dbSchema` 为空时，尝试通过数据库连接动态获取：
   ```go
   if dbSchema == "" {
       conn := admin.GetConnNoCheck(connID)
       dbSchema = getCurrentSchema(conn)
   }
   ```

## 九、潜在问题与改进建议

### 9.1 通用问题

1. **Agent 缓存 Key 不包含完整 schemas**：`agentCacheKey` 只用 `connID::dbSchema::userID`，如果用户在多 schema 模式下切换了 schemas 选择但主连接不变，会复用旧的 Agent（包括旧的 tools 和 connLookup 映射）。建议将 schemas 信息纳入缓存 key 或在 Agent 复用时更新 tools。

2. **PermissionScope 只基于主 schema**：`BuildPermissionScope(userId, connID, dbSchema)` 只检查一个 schema 的权限，多 schema 模式下其他 schema 的权限没有被纳入 scope。建议为每个选中的 schema 分别构建权限范围。

3. **读写权限未区分**：当前 `CheckTableWritePermission` 与 `CheckTablePermission` 逻辑相同，未区分读/写权限。如果需要更细粒度的控制，建议在 `t_power` 表中增加 `access_type` 字段（read/write/all）。

4. **密码哈希算法**：当前使用 MD5 + 固定 salt，建议升级为 bcrypt 或 argon2 等现代哈希算法。

5. **列级权限在经典模式下降级**：经典模式下 `GetTableAccessDowngraded` 将列级权限降级为表级权限，意味着拥有列级权限的用户在经典模式下可以看到整表数据。如果需要一致性，建议经典模式也支持列级过滤。

### 9.2 空 Schema 相关问题（详见第八章）

6. **`BuildPermissionScope` 空 schema 时放宽匹配**（🟡 中风险）：与 `checkPowerByParam` 的收紧策略不一致，仅在边缘场景（连接无法查询 schema 列表且 db_schema 为空）触发，但若触发则权限放大。建议统一为安全侧倒或动态获取 schema。

7. **前端 `parseSchemaValue` 可能产生空 schema 的 SchemaRef**（🟡 中风险）：`parseSchemaValue("connId::")` 返回 `{connId, schema: ""}`，绕过了 `schemas.length === 0` 的前端拦截。建议在 `buildRequestSchemas` 中过滤空 schema。
